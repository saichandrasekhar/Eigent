Core Concepts

How Eigent Thinks About Trust

Five primitives that together solve the agent identity problem: tokens, chains, intersection, revocation, and audit.

🔐 PRIMITIVE 01

Eigent Token

An Eigent token is a JWS (JSON Web Signature) that cryptographically binds an agent's identity to the human who authorized it. Unlike API keys or bearer tokens, an Eigent token carries its entire provenance: who created it, what it's allowed to do, and where it sits in the delegation chain.

Token Structure
Header
alg: ES256
typ: eigent+jwt
kid: registry key ID
Payload (Claims)
sub: agent identity
scope: [allowed actions]
parent: who delegated
chain: full ancestry
depth: hop count / max
Signature
ES256 (P-256 ECDSA)
Verifiable via registry
JWKS endpoint 
// Decoded Eigent Token
{
  // Header
  "alg": "ES256",
  "typ": "eigent+jwt",

  // Payload
  "sub":       "agent:code-reviewer",          // Agent identity
  "iss":       "https://registry.eigent.dev",   // Issuing registry
  "iat":       1711900800,                     // Issued at
  "exp":       1711987200,                     // Expires at
  "scope":     ["read_file", "run_tests"],    // Allowed actions
  "parent":    "human:alice@company.com",       // Direct parent
  "chain":     ["human:alice@company.com"],      // Full ancestry
  "depth":     1,                              // Current depth
  "max_depth": 5                               // Maximum allowed
}

Key properties:

  • Tokens are signed by the registry, not by the agent itself. This prevents token forgery.
  • The chain field traces all the way back to the human. No orphaned agents.
  • The scope field is an explicit allowlist. Anything not listed is denied by default.
  • Tokens expire. There are no permanent agent credentials.
🔗 PRIMITIVE 02

Delegation Chains

A delegation chain is the ordered sequence of trust relationships from a human to the agent performing an action. When Agent A delegates to Agent B, it creates a new link in the chain. The fundamental rule: permissions can only narrow, never widen.

Permission Narrowing at Each Hop
👤 Alice read, write, test, deploy
🤖 code-reviewer read, write, test
🤖 test-runner test
🚫 delete_file BLOCKED

Chain invariants:

  • Monotonic narrowing: child.scope ⊆ parent.scope at every hop
  • Depth limits: max_depth prevents unbounded chains (default: 5)
  • Ancestry preservation: The full chain is stored in the token, not just the direct parent
  • No cycles: An agent cannot appear twice in its own chain
🛠 PRIMITIVE 03

3-Way Scope Intersection

When the sidecar evaluates a tool call, it doesn't just check the agent's token. It performs a 3-way intersection of three independent permission sources. The agent can only perform an action if all three agree.

Effective Scope = Intersection of Three Sources
Human
Permissions
Agent
Capabilities
Resource
Sensitivity
Effective Scope
// 3-way intersection example

human_permissions  = ["read", "write", "test", "deploy"]
agent_token_scope  = ["read", "write", "test"]
resource_allows    = ["read", "test"]  // resource is read-only + testable

effective_scope    = human ∩ agent ∩ resource
                   = ["read", "test"]

// "write" is DENIED even though human and agent allow it,
// because the resource doesn't permit writes.
💥 PRIMITIVE 04

Cascade Revocation

When you revoke an agent's token, every agent it delegated to is also revoked, recursively. There are no orphaned sub-agents. Revoking the root human session kills the entire tree.

Before Revocation
👤 Alice
→
🤖 Agent A
→
🤖 Agent B
→
🤖 Agent C
After Revoking Agent A
👤 Alice
→
❌ Agent A
→
❌ Agent B
→
❌ Agent C
$ eigent revoke agent-a --cascade

❌ Revoked "agent-a"
   Cascade: also revoked 2 child agents
   - agent-b (depth 2)
   - agent-c (depth 3)
   Audit event: evt_revoke_cascade_f8a2c1

Revocation guarantees:

  • Revocation is immediate — the registry invalidates the token and all descendants
  • The sidecar checks revocation status on every request (online validation or CRL cache)
  • Revocation events are recorded in the audit trail with cascade details
  • Revoked tokens cannot be re-used even if the signature is still valid
📑 PRIMITIVE 05

Merkle-Chained Audit Trail

Every event in Eigent — token issuance, delegation, verification, allow, deny, revocation — is recorded as an entry in a Merkle-chained log. Each entry includes the hash of the previous entry, creating a tamper-evident chain. If any entry is modified or deleted, the hash chain breaks.

Merkle Chain Structure
EVENT 001
TOKEN_ISSUED
prev: 0x0000...
hash: 0x9f86...
→
EVENT 002
DELEGATION
prev: 0x9f86...
hash: 0x4e07...
→
EVENT 003
SCOPE_DENIED
prev: 0x4e07...
hash: 0xa3f2...
// Audit trail entry
{
  "id":         "evt_003_a3f2c1d8",
  "timestamp":  "2025-03-15T14:32:01.847Z",
  "type":       "SCOPE_DENIED",
  "agent":      "test-runner",
  "action":     "delete_file",
  "chain":      ["alice", "code-reviewer", "test-runner"],
  "detail":     "scope 'delete_file' not in [run_tests]",
  "prev_hash":  "sha256:4e07408...",
  "hash":       "sha256:a3f2c1d8...",
  "compliance": {
    "eu_ai_act": ["Art.9", "Art.12", "Art.14"],
    "soc2":      ["CC6.1", "CC7.2"]
  }
}

Audit properties:

  • Tamper-evident: Modifying any entry breaks the hash chain
  • Complete: Every action is logged, including denials
  • Compliance-mapped: Each event links to EU AI Act articles and SOC2 controls
  • Queryable: Filter by agent, time range, action type, or chain
